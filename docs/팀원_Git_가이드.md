## 📌 Git 핵심 개념

### Git이 파일을 지키는 원리 (3줄 요약)

1. **Git은 타임머신입니다** — 모든 변경이 기록되어서, 언제든 과거로 되돌릴 수 있습니다.
2. **잘못해도 복구됩니다** — `commit`(저장)한 것은 삭제해도 되살릴 수 있습니다.
3. **main은 보호됩니다** — main 브랜치에는 직접 push할 수 없게 잠겨 있어서, 실수로 망가뜨릴 수 없습니다.

> 💡 **쉽게 말하면**: Word의 "되돌리기(Ctrl+Z)"가 영원히 남아있는 것이 Git입니다.

### Git의 4가지 공간

Git에는 코드가 이동하는 **4가지 공간**이 있습니다. 모든 명령어는 이 공간 사이에서 코드를 옮기는 것입니다.

```
┌──────────────┐    git add     ┌──────────────┐   git commit   ┌──────────────┐   git push    ┌──────────────┐
│              │ ─────────────→ │              │ ─────────────→ │              │ ────────────→ │              │
│  작업 디렉토리  │               │  스테이징 영역  │               │   로컬 저장소   │              │  원격 저장소   │
│ (내 컴퓨터)    │ ←───────────── │  (커밋 대기실)  │               │  (.git 폴더)  │ ←──────────── │  (GitHub)    │
│              │  git checkout  │              │               │              │   git pull    │              │
└──────────────┘               └──────────────┘               └──────────────┘              └──────────────┘

  📂 내가 직접         📦 "이 파일들을           💾 세이브 포인트가        ☁️ 팀원 모두가
  수정하는 파일들        커밋할 거야" 표시          저장되는 곳              공유하는 저장소
```

### 브랜치란?

브랜치는 **독립적인 작업 공간**입니다. 게임의 세이브 슬롯처럼, 각자의 브랜치에서 작업하면 서로의 코드에 영향을 주지 않습니다.

```
main ─────●─────●─────●─────●─────●───── (안정된 코드, 보호됨)
           \                     ↑
            \                   머지 (PR)
             \                 /
  feature/   ●────●────●────● (김민규의 작업 공간)
  김민규-로그인
```

### 포크(Fork)란?

포크는 **조직 레포를 내 GitHub 계정으로 통째로 복사**하는 것입니다. 내 복사본에서 자유롭게 작업한 뒤, 조직 레포에 PR을 보냅니다.

```
┌─────────────────────────┐         PR (Pull Request)         ┌─────────────────────────┐
│  monglepick/backend     │ ◄──────────────────────────────── │  yhj0904/backend        │
│  (조직 레포, 원본)        │                                    │  (내 포크, 복사본)         │
│                         │ ──────────────────────────────────→│                         │
│  origin                 │         fork (처음 1번)             │  fork (또는 origin)      │
└─────────────────────────┘                                    └─────────────────────────┘
```

## 📖 Git 명령어 사전 (알아야 할 명령어 모음)

### 🟢 매일 쓰는 명령어 (필수!)

| 명령어 | 설명 | 비유 | 예시 |
|---|---|---|---|
| `git status` | 현재 상태 확인 (뭐가 바뀌었는지) | 📋 현재 상태 체크리스트 | `git status` |
| `git add` | 파일을 커밋 대기열에 추가 | 📦 택배 상자에 물건 넣기 | `git add LoginController.java` |
| `git commit` | 변경사항을 저장 (세이브 포인트) | 💾 게임 세이브 | `git commit -m "✨ 로그인 구현"` |
| `git push` | 로컬 커밋을 GitHub에 업로드 | ☁️ 클라우드에 업로드 | `git push origin feature/김민규-로그인` |
| `git pull` | GitHub에서 최신 코드 다운로드 | ⬇️ 클라우드에서 다운로드 | `git pull origin main` |
| `git checkout` | 브랜치 이동 또는 파일 복구 | 🚪 다른 방으로 이동 | `git checkout main` |
| `git checkout -b` | 새 브랜치 만들고 이동 | 🏗️ 새 방 만들고 입장 | `git checkout -b feature/김민규-로그인` |

### 🟡 가끔 쓰는 명령어 (알아두면 유용)

| 명령어 | 설명 | 비유 | 예시 |
|---|---|---|---|
| `git branch` | 브랜치 목록 확인 | 📂 세이브 슬롯 목록 | `git branch` |
| `git branch -d` | 머지된 브랜치 삭제 | 🗑️ 다 쓴 세이브 슬롯 삭제 | `git branch -d feature/김민규-로그인` |
| `git log` | 커밋 이력 확인 | 📜 히스토리 보기 | `git log --oneline -10` |
| `git diff` | 변경 내용 비교 | 🔍 뭐가 바뀌었는지 상세 보기 | `git diff` |
| `git stash` | 변경사항 임시 저장 | 🧊 냉동 보관 | `git stash` |
| `git stash pop` | 임시 저장 꺼내기 | 🔥 냉동 해동 | `git stash pop` |
| `git merge` | 다른 브랜치 코드 합치기 | 🔀 두 길을 하나로 합치기 | `git merge main` |
| `git revert` | 커밋을 안전하게 취소 | ⏪ 되감기 (기록 남김) | `git revert HEAD` |
| `git clone` | 레포를 처음 복사 | 📥 처음 다운로드 | `git clone https://github.com/monglepick/...` |
| `git remote -v` | 연결된 원격 저장소 확인 | 🔗 연결 상태 확인 | `git remote -v` |

### 🔴 알아는 두되 조심해야 할 명령어

| 명령어 | 설명 | 위험도 | 언제 쓰나? |
|---|---|---|---|
| `git reset --soft HEAD~1` | 마지막 커밋 취소 (코드는 유지) | ⚠️ 주의 | main에 실수로 커밋했을 때 |
| `git checkout -- 파일명` | 특정 파일 수정 내용 되돌리기 | ⚠️ 주의 | 수정한 파일 하나를 원래대로 |
| `git remote add` | 원격 저장소 연결 추가 | 안전 | 포크 방식에서 조직 레포 연결할 때 |

## 🅰️ 워크플로우 A: 브랜치 방식 (간단) ⭐ 추천

> **조직 레포에서 직접 브랜치를 만들어 작업하는 방식입니다.**
> remote가 1개(origin)라 헷갈릴 일이 없습니다.

### 전체 흐름도

```
  조직 레포 (monglepick/monglepick-backend)
  ─────────────────────────────────────────
  main ─────●─────●─────●──────────●─────●──── (보호됨, 직접 push 불가)
             \                    ↑
              \ checkout -b      │ PR 머지
               \                 │
  feature/     ●────●────●────●──┘
  김민규-로그인    add   commit  push
```

### 처음 1번: 레포 복제

```bash
# 조직 레포를 내 컴퓨터로 복제합니다 (처음 1번만!)
git clone https://github.com/monglepick/monglepick-backend.git

# 폴더로 이동합니다
cd monglepick-backend
```

### 매일 작업 흐름

```bash
# ═══════════════════════════════════════
# 🟢 STEP 1. 작업 시작 전 — main 최신화 + 브랜치 생성
# ═══════════════════════════════════════

git checkout main                              # main 브랜치로 이동
git pull origin main                           # 최신 코드 가져오기
git checkout -b feature/김민규-로그인기능         # 새 브랜치 만들고 이동

# ═══════════════════════════════════════
# 🟡 STEP 2. 작업 중 — 수정하고 자주 커밋 (반복!)
# ═══════════════════════════════════════

git status                                     # 변경 파일 확인
git add src/main/java/.../LoginController.java # 파일 스테이징
git commit -m "✨ 로그인 API 구현"               # 커밋 (세이브!)

# ═══════════════════════════════════════
# 🔵 STEP 3. 작업 끝 — push 하고 PR 만들기
# ═══════════════════════════════════════

git push origin feature/김민규-로그인기능         # GitHub에 올리기
# → GitHub 웹에서 "Compare & pull request" 클릭 → PR 생성 → 머지

# ═══════════════════════════════════════
# ✅ STEP 4. 다음 작업 준비
# ═══════════════════════════════════════

git checkout main                              # main으로 복귀
git pull origin main                           # 머지된 최신 코드 반영
git checkout -b feature/김민규-회원가입           # 다음 작업 브랜치 생성
```

### Remote 구조 (1개)

```
origin → monglepick/monglepick-backend (조직 레포)
```

## 🅱️ 워크플로우 B: 포크 방식 (고급)

> **조직 레포를 내 계정으로 포크(복사)한 뒤, 내 포크에서 작업하고 조직에 PR을 보내는 방식입니다.**
> remote가 2개(origin + fork)라 좀 더 복잡하지만, 조직 레포를 더 깔끔하게 관리할 수 있습니다.

### 전체 흐름도

```
  조직 레포 (monglepick/backend) ← "origin"
  ────────────────────────────────────────
  main ─────●─────●─────●──────────●───── (보호됨)
                                   ↑
                                   │ PR 머지 (포크 → 조직)
                                   │
  내 포크 (yhj0904/backend) ← "fork"
  ────────────────────────────────────────
  main ─────●─────●─────●
             \
              \  checkout -b
               \
  feature/     ●────●────●────● → push to fork
  형주-작업                         → PR to origin
```

### 처음 1번: 포크 + 클론 + remote 설정

```bash
# 1) GitHub 웹에서 조직 레포의 "Fork" 버튼을 클릭합니다
#    → 내 계정에 복사본이 생깁니다 (예: yhj0904/monglepick-backend)

# 2) 조직 레포를 내 컴퓨터로 복제합니다
git clone https://github.com/monglepick/monglepick-backend.git
cd monglepick-backend

# 3) 내 포크를 "fork"라는 이름으로 연결합니다
git remote add fork https://github.com/yhj0904/monglepick-backend.git

# 4) 확인: remote가 2개여야 합니다
git remote -v
# origin  https://github.com/monglepick/monglepick-backend.git (fetch)   ← 조직 레포
# origin  https://github.com/monglepick/monglepick-backend.git (push)
# fork    https://github.com/yhj0904/monglepick-backend.git (fetch)      ← 내 포크
# fork    https://github.com/yhj0904/monglepick-backend.git (push)
```

### 매일 작업 흐름

```bash
# ═══════════════════════════════════════
# 🟢 STEP 1. main 최신화 (조직 레포에서 가져오기)
# ═══════════════════════════════════════

git checkout main                              # main 브랜치로 이동
git pull origin main                           # 조직 레포의 최신 코드 가져오기
git checkout -b feature/형주-작업내용             # 새 브랜치 만들기

# ═══════════════════════════════════════
# 🟡 STEP 2. 작업 중 — 수정하고 자주 커밋 (동일)
# ═══════════════════════════════════════

git status                                     # 변경 파일 확인
git add 파일이름                                 # 파일 스테이징
git commit -m "✨ 작업 내용"                      # 커밋

# ═══════════════════════════════════════
# 🔵 STEP 3. 작업 끝 — ⚠️ fork로 push!
# ═══════════════════════════════════════

git push fork feature/형주-작업내용               # ← 여기가 다릅니다! origin이 아니라 fork!

# → GitHub에서 PR 생성 시:
#   base: monglepick/monglepick-backend (main)  ← 조직 레포
#   head: yhj0904/monglepick-backend (feature/형주-작업내용)  ← 내 포크
# → PR 머지 후 조직 레포의 main에 반영됩니다

# ═══════════════════════════════════════
# ✅ STEP 4. 다음 작업 준비 (동일)
# ═══════════════════════════════════════

git checkout main
git pull origin main                           # 항상 조직(origin)에서 pull
```

### Remote 구조 (2개)

```
origin → monglepick/monglepick-backend (조직 레포) ← pull은 여기서
fork   → yhj0904/monglepick-backend   (내 포크)   ← push는 여기로
```

### 브랜치 vs 포크 — 뭐가 다른가요?

| | 브랜치 방식 (A) | 포크 방식 (B) |
|---|---|---|
| **난이도** | ⭐ 쉬움 | ⭐⭐⭐ 어려움 |
| **remote 수** | 1개 (origin) | 2개 (origin + fork) |
| **push 대상** | `git push origin ...` | `git push fork ...` |
| **조직 레포** | 브랜치가 많아질 수 있음 | 깔끔하게 유지됨 |
| **추천 대상** | Git 초보, 소규모 팀 | Git 익숙한 사람, 오픈소스 |
| **우리 팀 추천** | ⭐ **김민규, 이민수, 정한나** | 윤형주 |

> 💡 **어떤 방식이든 PR로 main에 머지하는 것은 동일합니다.** 차이는 push할 때 `origin`으로 하느냐 `fork`로 하느냐뿐입니다.

### 전체 의사결정 흐름

```

                              │   작업 시작   │
                              └──────┬──────┘
                                     │
                                     ▼
                         ┌───────────────────────┐
                         │ git checkout main      │
                         │ git pull origin main   │
                         └───────────┬───────────┘
                                     │
                                     ▼
```

## ❓ 이런 상황엔 이렇게 (FAQ)

### "실수로 파일을 삭제했어요!"

걱정 마세요. 커밋한 적이 있다면 복구할 수 있습니다.

```bash
# 삭제한 파일을 마지막 커밋 상태로 복구합니다
git checkout -- 삭제한파일.java

# 예시: UserController.java를 복구
git checkout -- src/main/java/com/monglepick/domain/user/controller/UserController.java
```

아직 커밋하기 전이라면:
```bash
# 모든 변경사항을 마지막 커밋 상태로 되돌립니다 (주의: 수정사항도 사라집니다)
git checkout -- .
```

---

### "커밋을 잘못했어요!"

```bash
# 방금 한 커밋을 취소하는 새 커밋을 만듭니다 (안전한 방법!)
git revert HEAD

# 에디터가 열리면 :wq 입력하고 Enter (또는 그냥 저장하고 닫기)
```

> ✅ `git revert`는 이력이 남으면서 취소됩니다. 안전합니다!
>
> ❌ `git reset --hard`는 쓰지 마세요! (이력이 사라집니다)

---

### "충돌(conflict)이 났어요!"

충돌은 **같은 파일의 같은 부분**을 두 사람이 수정했을 때 발생합니다. 무서운 게 아닙니다!

```bash
# 1) 어떤 파일에서 충돌이 났는지 확인합니다
git status
```

충돌 파일을 열면 이렇게 보입니다:
```
<<<<<<< HEAD
내가 수정한 코드
=======
다른 사람이 수정한 코드
>>>>>>> origin/main
```

**해결 방법**:
1. `<<<<<<<`, `=======`, `>>>>>>>` 표시를 지웁니다
2. 두 코드 중 맞는 것을 남기거나, 합칩니다
3. 저장하고 커밋합니다

```bash
# 충돌 해결 후
git add 충돌해결한파일.java
git commit -m "🔀 충돌 해결"
```

> 💡 **VS Code 팁**: 충돌이 나면 VS Code에서 "Accept Current Change" / "Accept Incoming Change" / "Accept Both Changes" 버튼이 나타납니다. 클릭 한 번으로 해결할 수 있습니다!

---

### "push가 안 돼요!"

```bash
# 원격에 새로운 변경사항이 있으면 먼저 가져와야 합니다
git pull origin 내브랜치명

# 예시
git pull origin feature/김민규-로그인기능

# 그 다음 다시 push
git push origin feature/김민규-로그인기능
```

> 만약 `git pull` 할 때 충돌이 나면, 위의 "충돌이 났어요" 섹션을 참고하세요.

---

### "뭘 했는지 모르겠어요!"

```bash
# 현재 어떤 브랜치에 있고, 뭐가 변경됐는지 확인합니다
git status

# 지금까지의 커밋 이력을 봅니다 (q를 누르면 나갑니다)
git log --oneline -10

# 어떤 브랜치들이 있는지 확인합니다 (* 표시가 현재 브랜치)
git branch
```

---

### "잘못된 브랜치에서 작업했어요!"

```bash
# 아직 커밋하지 않은 변경사항을 임시 저장합니다
git stash

# 올바른 브랜치로 이동합니다
git checkout feature/김민규-로그인기능

# 임시 저장한 변경사항을 가져옵니다
git stash pop
```

---

### "main에 커밋해버렸어요!"

main에 직접 push는 못하도록 막혀있으니, push만 안 했다면 괜찮습니다.

```bash
# 1) 새 브랜치를 만듭니다 (커밋이 이 브랜치에 복사됩니다)
git branch feature/김민규-실수구제

# 2) main을 원래 상태로 되돌립니다 (1개 커밋 전으로, 코드는 유지)
git reset --soft HEAD~1

# 3) 새 브랜치로 이동합니다
git checkout feature/김민규-실수구제

# 4) 거기서 커밋하고 push 합니다
git add .
git commit -m "✨ 내 작업 내용"
git push origin feature/김민규-실수구제
```

## 🚫 절대 쓰지 마세요! (위험 명령어)

| 명령어 | 왜 위험한가요? | 대신 이걸 쓰세요 |
|---|---|---|
| `git push --force` | 원격 저장소의 다른 사람 코드가 날아갑니다 | `git pull` 후 `git push` |
| `git push -f` | 위와 동일 (줄임말) | `git pull` 후 `git push` |
| `git reset --hard` | 커밋하지 않은 수정사항이 전부 사라집니다 | `git stash` (임시 저장) |
| `git clean -f` | 추적되지 않는 파일이 전부 삭제됩니다 | 수동으로 불필요한 파일 삭제 |
| `git checkout .` | 수정한 내용이 전부 사라집니다 | 필요한 파일만 `git checkout -- 파일명` |
| `git branch -D` | 브랜치가 강제 삭제됩니다 | `git branch -d` (머지된 것만 삭제) |

> 🛡️ **안심하세요**: 우리 프로젝트에는 안전 훅(hook)이 설치되어 있어서, `git push --force`나 main 직접 push를 시도하면 **자동으로 차단**됩니다.

## 💻 VS Code에서 Git 사용하기 (터미널이 어렵다면)

터미널 명령어가 어렵다면, VS Code의 GUI로도 똑같은 작업을 할 수 있습니다!

### 1. 브랜치 만들기/바꾸기

- VS Code 왼쪽 하단의 **브랜치 이름**을 클릭합니다
- "Create new branch..." 선택 → 브랜치명 입력 → Enter

### 2. 변경사항 확인 + 커밋

- 왼쪽 사이드바에서 **소스 제어(Source Control)** 아이콘을 클릭합니다 (가지 모양, 단축키: `Ctrl+Shift+G`)
- 변경된 파일 목록이 보입니다
- 파일 옆의 **+** 버튼 → 스테이징(add)
- 상단 메시지 입력창에 커밋 메시지 작성 → **✓ 체크** 버튼(또는 `Ctrl+Enter`) → 커밋

### 3. Push

- 소스 제어 패널의 **···** (점 세개) → **Push** 클릭
- 또는 왼쪽 하단 브랜치명 옆의 **↑** 화살표 클릭

### 4. Pull

- 소스 제어 패널의 **···** → **Pull** 클릭
- 또는 왼쪽 하단 브랜치명 옆의 **↓** 화살표 클릭

> 💡 **추천 확장 프로그램**: **GitLens** 를 설치하면 누가 어떤 코드를 언제 수정했는지 한눈에 볼 수 있습니다.
